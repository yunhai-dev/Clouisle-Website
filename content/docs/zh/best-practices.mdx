## 功能概述

最佳实践不是额外功能，而是一套帮助你少走弯路的落地方法。  
它的作用是把“能用”尽快变成“稳定、可维护、可扩展”。

## 适用场景

适合：

- 第一次搭建核心能力
- 准备从试点转向正式上线
- 想降低返工、幻觉、越权和成本失控风险

## 上线前建议按这个顺序检查

### 第 1 步：先把 Agent 做小，而不是做全

一开始最容易犯的错误，是试图让一个 Agent 同时解决太多事情。  
更稳妥的做法是：

- 一次只解决一个核心业务目标
- 先让基础对话稳定
- 再逐步增加知识、工具或记忆能力

### 第 2 步：先把知识库做准，而不是做大

知识库建设的关键顺序应是：

1. 先保证资料质量
2. 再导入样本做测试
3. 再逐步扩大资料规模

如果一开始只追求文档数量，后面调优成本会非常高。

### 第 3 步：先把工作流主干跑通，再补复杂分支

工作流最稳的推进方式通常是：

- 先写清主干流程
- 先跑最短成功路径
- 再补条件分支、循环和异常处理

这样做比一开始把整条大流程全部铺满更容易成功。

### 第 4 步：先建立治理边界，再扩大使用范围

当平台开始多人使用后，应优先建立：

- 角色与权限边界
- 团队归属
- API Key 规范
- 模型授权策略

如果这些边界建立得太晚，后续平台会越来越难治理。

### 第 5 步：把测试样例和回归习惯沉淀下来

无论是 Agent、知识库还是工作流，建议都保留：

- 固定测试问题
- 固定输入样例
- 修改后的回归检查清单

这样后续迭代时才能知道效果到底是变好还是变差。

## 结果验证

最佳实践真正落地时，通常会表现为：

- 返工次数减少
- 故障定位更快
- 新成员更容易接手
- 平台能力越做越多，但复杂度没有失控

## 常见问题

### 为什么团队越努力做功能，反而越乱

通常是因为缺少统一方法，大家都在并行加功能，但没有稳定的推进顺序和回归习惯。

### 为什么“先小后大”这么重要

因为平台能力之间耦合很高。  
一开始做得太大，问题会同时出现在模型、知识、工具和权限多个层面。

### 为什么最佳实践要写成固定规则

如果只停留在个人经验，团队一扩大就会失效。  
只有形成固定规则，平台能力才能持续交接和复用。

## 注意事项

- 最佳实践的重点是稳定性，不是功能堆叠
- 一旦团队形成有效方法，应尽快固化成内部标准
- 每次复盘后，都建议把新经验补充回这套方法里
