## 功能概述

模型决定 Agent 和工作流“会不会想”，工具决定它们“能不能做”。  
这两类能力共同构成平台的基础能力层。

![模型中心](/images/docs/models-center.png)
![工具中心](/images/docs/tools-center.png)

## 适用场景

这一部分适合：

- 准备接入新的模型提供商
- 准备给 Agent 增加系统调用能力
- 需要统一治理成本、权限和可用性

## 前置条件

开始前建议准备：

- 可用的模型凭证
- 工具目标系统的调用信息或凭证
- 对哪些团队、应用要开放这些能力的计划

## 操作步骤

### 第 1 步：先接模型，再接工具

模型是基础前提。  
如果模型层不稳定，后面工具和工作流测试都会受到影响。

### 第 2 步：完成模型中心基础配置

重点通常包括：

- 提供商
- Base URL 和 API Key
- 模型规格
- 能力标签
- 默认参数

### 第 3 步：再配置工具中心

工具配置通常包括：

- 工具类型
- 请求参数或命令参数
- 认证方式
- 返回值解析

如果你需要从治理视角统一查看工具，也可以进入后台 `工具管理` 页面，先确认当前有哪些工具、属于什么类型、处于什么状态。

![后台工具管理页](/images/docs/admin-tools-management.png)

![创建工具菜单](/images/docs/tool-create-menu.png)
![创建 HTTP 工具表单](/images/docs/tool-http-api-dialog.png)

如果选择 `HTTP API` 类型，通常需要先填写：

- 工具名称和显示名称
- 描述与分类
- 请求方法和目标 URL
- 请求头、查询参数和超时时间
- 入参定义，方便在 Agent 或工作流中动态传值

对于平台内置工具，还可能出现“需要配置”的状态。  
这时通常不是工具不可用，而是还没有补齐对应的外部凭证。

![内置工具凭证配置弹窗](/images/docs/tool-credential-config-dialog.png)

例如网页搜索工具会要求先填写对应服务的 API Key。只有这一步完成后，工具测试和 Agent 调用才会真正可用。

### 第 4 步：在真实应用里联动验证

最后不要停留在模型中心或工具中心里单独测试。  
应把它们接入 Agent 或工作流，确认真实链路可以跑通。

如果是工具能力，建议先在工具弹窗里直接做一次最小测试，确认：

- 参数能否正常输入
- 返回结果是否符合预期
- 后续是否适合继续接到 Agent 或工作流里

![工具测试弹窗](/images/docs/tool-web-search-test-dialog.png)

## 结果验证

建议至少确认：

- 模型测试可以正常返回
- 工具调用成功且返回值结构可消费
- Agent 或工作流中能正确引用这些能力

## 价值说明

模型与工具治理到位后，平台才能同时具备：

- 能力上限
- 可执行性
- 可复用性
- 可治理性

## 注意事项

- 新模型和新工具都应先在小范围验证，再逐步开放
- 工具返回结构不稳定时，优先解决解析问题，不要只调提示词
- 能力层越丰富，越需要统一命名和状态治理
