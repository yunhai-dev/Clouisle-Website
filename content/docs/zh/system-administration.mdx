## 功能概述

系统管理面向平台治理，而不是单次业务使用。  
它负责让管理员回答下面这些关键问题：

- 现在平台运行得怎么样
- 谁能访问平台
- 团队和权限是否清晰
- 安全、通知和审计是否已经建立

## 适用场景

这一页适合：

- 平台准备正式对内或对外开放
- 管理员第一次接手后台
- 平台需要从试用状态走向长期运营

## 前置条件

开始前建议具备：

- 至少一个可用管理员账号
- 平台已经有基础用户或团队
- 已对登录、通知、权限和审计有初步规划

## 操作步骤

### 第 1 步：先进入后台仪表盘，确认平台现状

管理员进入后台后，第一眼先看仪表盘，而不是直接去改配置。  
重点确认：

- 当前有多少用户和团队
- 近期平台活动是否正常
- 是否存在明显异常波动

![后台仪表盘](/images/docs/dashboard-overview.png)

完成这一步后，你应该对平台当前规模和活跃度有一个基本认识。

### 第 2 步：再检查用户和团队结构

确认平台现状后，再进入用户和团队相关页面，重点看：

- 用户数量和状态是否正常
- 团队划分是否清晰
- 是否已经存在多团队协作需求

![后台用户管理页](/images/docs/admin-users.png)
![后台团队管理页](/images/docs/admin-teams.png)

这一步的目标是先把“人”和“组织”看清楚，再继续谈权限和规则。

### 第 3 步：梳理角色、权限和访问规则

当用户和团队结构清楚后，再确认：

- 角色是否已经建立
- 高风险权限是否被收紧
- 非管理员是否看不到站点级入口

如果这一步跳过，后续再配通知、SSO 或 API Key 都会留下治理漏洞。

### 第 4 步：补齐站点级设置

组织和权限稳定后，再去处理平台级规则，例如：

- 站点名称和基础信息
- 注册与登录策略
- 通知渠道
- SSO 规则
- 审计保留和告警阈值

建议站点级规则在正式开放前就一次性梳理清楚，不要边开放边频繁变更。

### 第 5 步：建立后台的日常巡检习惯

后台真正发挥作用，不只是把页面都配置完，而是形成固定巡检动作。  
建议至少建立下面几项例行检查：

- 看仪表盘整体趋势
- 看用户、团队和权限是否有异常变化
- 看审计日志里是否有高风险操作
- 看通知和告警链路是否正常

## 结果验证

一个治理基本完整的平台后台，至少应满足：

- 管理员可以顺畅进入核心后台页面
- 非管理员不会看到站点级高风险入口
- 用户、团队、权限和站点规则之间没有明显冲突
- 仪表盘、通知和审计链路已经可用

## 常见问题

### 为什么平台功能很多，但管理还是很乱

通常不是因为模块不够，而是因为：

- 没先梳理用户和团队边界
- 权限和站点策略建立得太晚
- 后台没有形成固定巡检机制

### 为什么后台第一步建议先看仪表盘

因为仪表盘能帮助你快速判断平台是否处于稳定状态。  
如果一上来就直接改设置，容易在不了解现状的情况下扩大问题。

### 为什么系统管理不能只做一次配置

因为平台在持续变化。  
只做一次配置不做持续巡检，治理能力很快就会失效。

## 注意事项

- 后台策略修改会影响全站，尽量在低风险时段执行
- 先看现状，再改规则，不要颠倒顺序
- 重大配置变更后，应立即做一次后台回归验证
